### 2) Blocks dropdown

**Backend work:** Optional slim endpoint, otherwise done.

`BlocksModel.favourite` and `BlocksModel.icon` already exist.

`GET /model/<int:model_id>/blocks` returns everything via `block.json(scenario_id)` — heavier than needed, but it already contains every field required by the dropdown.

#### Gap
The existing endpoint runs scenario compute. For a dropdown, you only need:

```json
[{ "id": ..., "name": ..., "icon": ..., "favourite": ..., "model_category_id": ... }]
```

#### Recommendation
Add `?fields=summary` on `BlockList.get` (or a new `BlockSummaryList` resource) that returns this shape **without touching scenarios**.

This same endpoint can also power the **Favourite Blocks** section below.

## Items in the Figma that go beyond the ticket

These appear in the Figma but are **not listed in the ticket’s 5 bullets**. Confirm scope with PM before building them.

### A) Switch-Model dropdown (“Favourite Models”)

Requires:

```json
[{ "id": ..., "name": ..., "date_created": ..., "block_count": ..., "dimension_count": ..., "is_favourite": ... }]
```

for each model the user can access.

#### Backend gaps
- No `favourite` flag exists on `ModelsModel`
- Need either:
  - a new `user_favourite_models (user_id, model_id)` junction table, or
  - an `is_favourite` column  
    - but since this is per-user, it really should be a junction table

#### Other required backend work
- Counts:
  - `len(model.blocks)`
  - `len(model.dimensions)`
- These can be added to `model.json()` or a new `/user/models/summary` endpoint
- Toggle endpoint:
  - `POST /model/<id>/favourite`
  - `DELETE /model/<id>/favourite`
